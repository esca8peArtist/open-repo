## Appendix: Troubleshooting by Red Flag

### If Metric 1 Fails (Signal Generation = 0)

**Symptom**: All 5 sessions show 0 non-HOLD signals by 20:00 UTC

**Check list** (in order):
1. ☐ HMM regime initialized? (Metric 2) — if regime=None, root cause is HMM priming, not signal generation
2. ☐ Models loaded correctly? `docker logs stockbot | grep -i "model.*loaded"` — if missing, check `/opt/stockbot/models/` directory
3. ☐ Feature computation running? `docker logs stockbot | grep "features computed"` — if missing, feature engineering blocked
4. ☐ Alpaca bar feed present? `docker logs stockbot | grep "bar received"` — if missing, data stream is starved

**Likely root cause**: HMM regime=None (see Metric 2) — bars not fed to HMM properly  
**Next step**: Investigate G6 (HMM initialization) and G3 (WebSocket singleton)

---

### If Metric 2 Fails (Regime=None)

**Symptom**: All 5 sessions show regime=None throughout market hours

**Check list**:
1. ☐ WebSocket singleton (G3)? If count >1, multiple streams created; if 0, no stream at all
2. ☐ WebSocket auth (G4)? If 406 errors present, connection limit exceeded
3. ☐ Bar count fed to HMM? `docker logs stockbot | grep "HMM bars_seen"` — should be >60 by 14:00 UTC
4. ☐ Timestamp passing in bars? June 22 fix added `timestamp=idx` to `masker.update_price()` — verify in source code

**Likely root cause**: 
- June 18 failure: WebSocket 406 (connection limit) → bars not received
- If timestamps not passed: deduplication blocks bars 2–82, only 1 bar fed to HMM

**Next step**: 
- If 406 present: May need WebSocket multiplexing fix
- If timestamp issue: Check commit 9194e6b (June 22 fix); if reverted, re-apply

---

### If Metric 4 Fails (Credential/Duplicate Errors)

**Symptom**: 40010001 errors or 401/403 during order submission

**Check list**:
1. ☐ API keys valid? `echo $ALPACA_API_KEY | wc -c` should be 40 chars (SK key length)
2. ☐ Account is paper, not live? Check Alpaca dashboard — "Paper Trading" label should be visible
3. ☐ Order ID deduplication working? `docker logs stockbot | grep "order_id"` — should show unique IDs
4. ☐ Alpaca API rate limit hit? `docker logs stockbot | grep "429"` — if present, throttle order submission

**Likely root cause**: Alpaca API key misconfiguration or rate limiting  
**Next step**: Verify `.env` on Jetson (`ssh awank@100.120.18.84 "cat /opt/stockbot/.env | grep ALPACA"`)

---

### If Metric 5 or 6 Fail (Drift/Alignment)

**Symptom**: Z-score >0.5 or model alignment <75%

**Check list**:
1. ☐ One day only: Day 1 metrics are baseline only; not statistically significant until day 5
2. ☐ Market conditions unusual? Check SPY close for the day; if >2% move, baseline drift expected
3. ☐ Model retraining needed? If live Sharpe <0.5 by day 5, schedule retraining on updated feature data

**Likely root cause**: Market volatility or insufficient baseline days  
**Next step**: Continue monitoring through day 5 (June 28); re-assess after 5 trading days

---

### If Metric 7 Fails (Regime Churn)

**Symptom**: >4 regime flips per session per hour

**Check list**:
1. ☐ HMM threshold set correctly? Verify `hmm_regime_masking: true` in session config
2. ☐ Bar timestamps consistent? If timestamps jump around, HMM sees artificial state changes
3. ☐ Market volatility spike? Check VIX for the day; high VIX can cause rapid regime switches

**Likely root cause**: Timestamp corruption or HMM configuration issue  
**Next step**: If persistent, add HMM logging to track state transitions; diagnose on June 25
