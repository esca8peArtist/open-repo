### Can All Three Projects Proceed June 1–15?

**Systems-resilience publication + Stockbot Lever B + Open-repo Phase 5.1: Yes, all three can proceed
simultaneously in June. One qualified caveat applies.**

The resources do not directly collide because each project uses different agent types:
- Stockbot Lever B uses a stockbot subagent (Jetson compute) plus bounded Orchestrator configuration decisions
- Open-repo Phase 5.1 is engineering-only (libzim wheel deployment, alembic upgrade, ZIM validation); 21 hours, engineering subagent only, no writing agent
- Systems-resilience publication uses the Orchestrator for editorial passes and a research agent for Phase 6 domain production

The one genuine bottleneck is Orchestrator coordination bandwidth in June 1–10. All four concurrent work
streams require Orchestrator decisions and setup in this same window. The JUNE_PHASE_6_RESOURCE_ALLOCATION.md
document estimated 18–25 Orchestrator coordination hours for June 1–15.

**Mitigation**: Batch all June 1 activation decisions into a single session. This converts four separate
coordination spikes into one 3–4 hour block. The June 1 session covers: Phase 6 domain briefing (2–3 hrs),
Lever B configuration decision (1–2 hrs), open-repo deployment confirmation (0.5–1 hr), Wave 1+2 editorial
kickoff (1–2 hrs). Total: 4.5–8 hours, concentrated in one session, not distributed across the week.

|---|---|---|---|

**The June 1–5 spike under Option A (21–33 hrs) is the most recoverable profile.** After June 5,
Option A Orchestrator demand drops sharply. Option B sustains 12–20 hrs/week through June 14. Option C
never drops below 10 hrs/week through July 4.

### Does One Project Need Deferral?

Under **Option A**: No deferral required for any of the three projects. The June 1 decision session
batches coordination needs. Stockbot (config-only path) is already resolved by June 1; retrain path
absorbs 5–8 hrs spread across 15 days. Open-repo engineering subagent is fully independent. Phase 6
research agent runs independently. The only true bottleneck is Orchestrator June 1–5 bandwidth (21–33
hrs in 5 days, approximately 4–7 hrs/day), which is above daily sustainable pace but bounded to five
calendar days.

Under **Option B**: No deferral strictly required, but the risk of needing to defer Phase 6 first-draft
reviews from June 14–18 to June 18–22 is meaningful if any June 1–14 event (Lever B retrain anomaly,
open-repo decision, author onboarding complexity) runs over. The single June 15 gate makes Option B
more sensitive to any deferred June 1–14 work.

Under **Option C**: No deferral strictly required, but the Phase 6 Domains E/F/H second-wave activation
effectively defers to July 10–15 (from June 20–25 under Options A/B) because Option C's rolling schedule
persistently occupies Orchestrator bandwidth. This is a soft deferral built into the option's structure.

|---|---|---|

genuine advantage (single-artifact authority) that is partially recoverable within Option A at the
cost of a 50-word "Volume 2" announcement in the Wave 1+2 release materials.**
