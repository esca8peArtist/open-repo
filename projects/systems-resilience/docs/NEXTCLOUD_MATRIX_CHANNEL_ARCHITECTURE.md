# Nextcloud+Matrix Channel & Folder Architecture for Wave 2

## Executive Summary

This document defines the complete folder structure in Nextcloud and room structure in Matrix for Wave 2 author collaboration (Domains 60-65, June 15 – July 15, 2026). Both platforms are configured to reflect a coherent information architecture: **per-domain isolated workspaces, shared resources, governance documentation, and cross-domain synthesis areas**.

The architecture follows these principles:
1. **Domain isolation**: Each domain has a private folder (Nextcloud) and private room (Matrix)
2. **Progressive visibility**: Shared resources are read-only to non-facilitators; governance is read-only to authors
3. **Sync integration**: Nextcloud file events trigger Matrix notifications (optional, Phase 6 enhancement)
4. **Redundancy**: If one platform fails, the other maintains operational capacity

### 1.2 Per-Domain Folder (One per Domain 60-65)

**Folder name:** `Phase5-Wave2-Domain-60-International-Coordination/` (for Domain 60; adjust for each domain)

**Members:** 3 authors (Tier A/B stratified) + 2 peer reviewers + 1 project lead (read-write)

**Subfolders and purpose:**

#### 1.2.1 `/drafts/`

**Purpose:** Active working folder for manuscript development

**Contents:**
- `outline.md` — Domain scope document (autogenerated from Item 67 brief template, read-only for authors)
- `annotated-bibliography.md` — 30-40 pre-vetted sources with quality ratings (read-only for authors)
- `Section-01-foundational-frameworks.md` — Author's working draft (writable by all 3 domain authors + peer reviewers)

## Part 3: Integration Points & Sync Architecture

### 3.1 Nextcloud→Matrix Notification Bridge (Optional, Phase 6 Enhancement)

**Use case:** When author uploads new section to `/domain-60/drafts/`, all peer reviewers and co-authors receive Matrix notification (without reading the file content, which stays in Nextcloud).

**Implementation (not required for June 15, but documented for future):**

```
Nextcloud event (file.created / file.updated)
    ↓
Webhook POST to external bridge
    ↓
Bridge parses event: /domain-60/drafts/Section-02-*.md created
    ↓
Matrix message posted to #domain-60:
    "Philip uploaded 'Section-02-case-studies.md' (2,100 words) at 15:30 UTC. Ready for peer review?"
```

**Webhook configuration (Nextcloud admin):**
- Settings → Administration → Webhooks
- URL: `https://matrix-bridge.internal/webhooks/nextcloud`
- Events: `file.created`, `file.updated`
- Filter: `/domain-*/drafts/`, `/domain-*/submitted/` (ignore `/published/` updates)

**Bridge script** (Python example):

```python
from flask import Flask, request
import requests

app = Flask(__name__)

@app.route('/webhooks/nextcloud', methods=['POST'])
def handle_nextcloud_event():
    data = request.json
    event_type = data.get('event')
    file_path = data.get('path')
    actor = data.get('actor')
    
    # Parse event
    if 'drafts' in file_path and '.md' in file_path:
        room_id = extract_domain_from_path(file_path)  # domain-60
        message = f"{actor} uploaded new section to {file_path}"
        
        # Post to Matrix room
        requests.post(
            f"https://matrix.example.com/_matrix/client/r0/rooms/{room_id}/send/m.room.message",
            json={"msgtype": "m.text", "body": message},
            headers={"Authorization": f"Bearer {BRIDGE_TOKEN}"}
        )
    
    return {"status": "ok"}, 200
```

**For June 15 deployment:** Webhook is optional (nice-to-have); core collaboration works without it. Implement in Phase 6 Week 2 if bandwidth allows.

## Part 5: Contingency Architecture (If One Platform Fails)

### 5.1 Nextcloud-Only Fallback (If Matrix Down)

**Timeline:** If Matrix unavailable >4 hours, activate email-based communication

**Workflow:**
- Authors post updates in `/domain-XX/drafts/comments.md` (text file)
- Peer reviewers check `/domain-XX/drafts/` folder daily (via Nextcloud sync client)
- Project lead sends email summary ("Domain 60 update: Philip completed Section 1-2; Natalia flagged source gap in Section 5...")
- Authors reply to email; project lead posts synthesis in `/Governance/decision-logs/`

**Timeline to restore Matrix:** Aim for <12 hours; otherwise switch to email-based coordination permanently

### 5.2 Matrix-Only Fallback (If Nextcloud Down)

**Timeline:** If Nextcloud unavailable >4 hours, activate offline-first mode

**Workflow:**
- Authors draft locally (Word, Google Docs, Markdown editor)
- Post drafts to Matrix as `.md` file attachments (message + attachment)
- Peer reviewers download attachment, review, post feedback in Matrix thread
- Authors re-upload revised version
- Collect all attachments in Matrix conversation history
- Once Nextcloud restored, bulk-import all attachments into appropriate folders

**Timeline to restore Nextcloud:** Aim for <24 hours; otherwise continue offline until July 10

### 5.3 Both Platforms Down (Contingency Offline Mode)

**Timeline:** If both Nextcloud + Matrix down >24 hours

**Activate:**
- Email-based coordination (all work via email attachments)
- Authors draft locally, email drafts to project lead
- Project lead archives in `~/projects/systems-resilience/wave-2-offline-archive/`
- Peer review via email reply-all
- Resume with Nextcloud/Matrix once restored

**Recovery workflow:** Bulk-import all email attachments once services restored

## Part 6: Moderation & Conflict Resolution

### 6.1 Domain Room Moderation

**Scope of moderation:** Keep conversation on-topic (domain research) and productive

**Off-topic examples (mild redirection):**
- Long political debate on unrelated topic → "Let's take this to #meta-governance if we want broader political discussion"
- Coordinating a local farm visit (nice, but off-topic) → "Great idea! Can we move this to #meta-governance or email? We want to keep domain rooms focused on research."

**Unacceptable (escalation to project lead):**
- Harassment or personal attacks
- Repeated off-topic spam
- Breach of confidentiality (sharing author drafts externally without permission)

**Peer reviewer response:**
1. If minor off-topic: Light redirection via message
2. If repeated: Mute user for 24 hours (prevents posting, see message history)
3. If serious: Kick user from room (can rejoin after project lead approval); escalate to project lead

### 6.2 Conflict Resolution (Author Disputes)

**Example disputes:**
- Author A: "The domain scope is unfair; it's 2x harder than other domains"
- Author B: "Peer reviewer gave contradictory feedback in reviews 1 and 2"
- Author C: "I didn't agree to having my draft shared in the cross-domain room"

**Process:**
1. **Informal resolution:** Project lead talks privately with author (Matrix DM or email)
2. **Document:** If dispute unresolved, log in `/Governance/decision-logs/conflict-resolution.md`
3. **Escalation:** If major (e.g., author wants to withdraw), escalate to project director (user) with full context

**No scope retroactively changed mid-project** (too disruptive); defer scope adjustments to Phase 7 or next wave
