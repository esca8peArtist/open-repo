## Part 3: Bottleneck Analysis

### Weeks with Multi-Domain Contention

**June 9–12 (most contested week in Batch 2)**

Three items compete in this window:
1. Domain 51 execution (user, 2.75 hours) — hard deadline July 1
2. Domain 49 Tier 2 sends under Scenario A/B (user, 1.0 hour)
3. Domain 39 Day 7 checkpoint (June 8 — immediately prior; monitoring signals affect Domain 51 pacing)

**Assessment**: User time is the constraint. Total user demand June 9–12 under Scenario A is 3.75 hours across 4 days — approximately 1 hour/day. This is achievable. The constraint is user willingness to execute on a daily cadence for 4 consecutive days. If user execution is batched (all sends on one day), there is no bottleneck.

**June 4–5 (domain 49 Scenario A spike)**

Under Scenario A, June 4–5 requires 4.0 user-hours in two days. This is the single largest concentrated user execution demand in all of Batch 2 for June. It is manageable but requires that the user be available for approximately 2 hours on each day, which is not guaranteed on short notice.

**July 1–5 (Domain 50 + Domain 48 overlap)**

Under all scenarios, July 1 has Domain 50 Gist creation and Lambda Legal/AT4E send. If Domain 48 is also activated in late June (Scenario A), it produces a late-June/July 1 overlap. Agent demand: 0.5–1.0 hours. User demand: 2.0–3.0 hours across July 1–5. Manageable with staggering (Domain 48 sends June 28–30; Domain 50 sends July 1–5).

## Part 4: June 10–15 Impact Assessment

### Stockbot Phase 2 Expansion (June 15 checkpoint, est. 76 hours)

**Conflict type**: Agent session competition
**Phase 2 active in this window**: Domain 51 distribution (June 9–12, 0.5 agent-hours) + Domain 49 Day 7 monitoring (June 11–12, 0 agent-hours)

**Assessment**: LOW conflict. Batch 2 requires fewer than 1 agent-hour in the June 9–15 window. Stockbot Phase 2 prep (76 hours, June 3–15) runs in dedicated infrastructure/engineering sessions. These are different session types that do not compete.

**User time conflict**: Stockbot June 15 checkpoint is passive monitoring — no user action required unless go/no-go is FAIL. Domain 51 sends (June 9–12) require 2.75 user-hours. These are independent: stockbot monitoring vs. email execution. No same-hour competition unless the June 15 checkpoint requires emergency infrastructure work, which would shift user priority.

**Conclusion**: Stockbot Phase 2 expansion does NOT delay Domain 51 or Domain 49 under any scenario. The single exception is if stockbot June 15 checkpoint is a hard FAIL requiring emergency infrastructure work — in that case, Domain 48 sends (optional, low urgency) defer, but Domain 51 proceeds because it has a July 1 hard deadline.

**Conclusion**: Systems-resilience Phase 5 Wave 2 does NOT delay any Batch 2 domain. The 120-hour allocation coexists with Batch 2 because they operate on different resource dimensions.
