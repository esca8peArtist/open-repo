## Template C: Peer Review Cycle (Per Domain)

*Use this template for each domain's peer review phase.*

### Peer Review Workflow

**Expert assigned**: [Name, Organization, Email]  
**Expertise**: [Methodology / Factual verification / Movement perspective / Policy window]  
**Expected review time**: [2-4 hours] (async)  

#### Handoff (June [date])

Email template:

```
Subject: [Domain X] Peer Review Request — [Title] (2-4 hour review, feedback by June [+2 days])

Dear [Expert Name],

We're producing a research domain on [Domain title] as part of Phase 2 of our 
institutional analysis project. We'd like your [expertise type] review of our draft.

**Domain scope**: [1-2 sentence summary]

**What we need**:
1. Factual accuracy check (are empirical claims supported by cited sources?)
2. [Methodology / Movement-perspective / Policy window] assessment
3. Any gaps or alternative framings you'd recommend

**Timeline**: 
- Draft attached below
- Your feedback by June [date]: ~2-4 hours of your time
- We'll implement revisions June [date], ready for publication June [date]

**Format**: Please return comments as:
- **Blocking issues** (P0): Factual errors or methodology concerns that must be fixed
- **Improvements** (P1): Alternative framings or additional sources worth considering
- **Nice-to-haves** (P2): Minor clarifications or expansions (defer if time-constrained)

[Draft pasted below]

Thank you for your time!
```

#### Feedback Reception & Logging (June [date+2])

1. **Read expert email** (10 min)
2. **Extract comments** into peer-review-comments.md file:
   ```markdown
   # Domain X Peer Review Comments
   
   **Expert**: [Name, Organization]
   **Date received**: June [date]
   **Review type**: [Type]
   **Total hours provided**: [2/3/4 hours]
   
   ## Blocking Issues (P0)
   - [Issue 1]: [Expert quote] → [Action: fix by June X]
   - [Issue 2]: [Expert quote] → [Action: fix by June X]
   
   ## Improvements (P1)
   - [Item 1]: [Expert suggestion] → [Decision: implement / defer]
   - [Item 2]: [Expert suggestion] → [Decision: implement / defer]
   
   ## Nice-to-Haves (P2)
   - [Item 1]: [Expert suggestion] → [Decision: defer]
   ```
3. **Prioritize fixes** (5 min): All P0 items must implement before publication. P1/P2 are discretionary.
4. **Implement P0 fixes** (hours vary): Update draft with critical fixes, track time.
5. **Verify expert acceptance** (optional, 30 min): Email expert confirming P0 fixes implemented + any major P1 items adopted. Ask if they want to spot-check revised sections.

#### Revision Priority Matrix

| Priority | Type | Implementation time | Definition | Publication rule |
|----------|------|-----------|-----------|---------|
| **P0** | Blocking | 30-120 min | Factual error, methodology flaw, missing critical source | **Must implement** before publication |
| **P1** | Improvement | 30-120 min | Alternative framing, additional source, movement-perspective insight | Implement if feasible within timeline; defer to patch if necessary |
| **P2** | Enhancement | 15-30 min | Minor clarification, additional context, editorial polish | **Can defer** to post-publication patch or next edition |

**Rule**: If P0 + P1 items total >3 hours, prioritize: (1) All P0 items, (2) P1 items with highest movement relevance, (3) defer low-priority P1.

## Template G: Contingency & Float Days

### Production Delay Risk Mitigation

**If peer review feedback reveals major revision needed** (>6 hours):
- Defer non-critical P1 items to post-publication patch
- Extend peer review-to-staging timeline by 1 day (June 15 → June 16)
- Notify user of revised publication date

**If source verification fails for Domain 57** (June 1-2 check):
- Substitute 1-2 sources (3 hours max)
- Use internet-archive proxy for any blocked sources
- Domain 57 timeline unchanged; work absorbs into existing buffer

**If user strategic review uncovers major repositioning need** (June 13):
- Implement high-priority revisions only (P0 + critical messaging changes)
- Defer lower-priority customizations to Tier 2 customization phase post-publication
- Maintain June 15 publication target

### Float Days Built Into Schedule

- **Week 1-2**: 2-hour buffer per domain (10% of estimated hours)
- **Week 3**: 1-day float if any domain runs over (June 12-13 contingency work day)
- **Publication staging**: 1-day buffer (June 15-16) if QA discovers unexpected issues

**Total buffer**: 3-4 days across 15-day production window (20% margin).
