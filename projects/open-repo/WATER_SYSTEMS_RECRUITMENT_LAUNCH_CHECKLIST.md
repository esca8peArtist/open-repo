- [ ] **Check 1.5 — Unsubscribe mechanism included**

  CAN-SPAM Act (US) and CASL (Canada) require a clear opt-out mechanism for commercial-adjacent email solicitations. For a volunteer recruitment email this is a best practice if not strict legal requirement, but the omission signals professionalism.

  Item 41 approach: The GitHub Issue is a natural opt-out mechanism. Include one of the following in each email footer:

  Option 1 (recommended): "If you'd prefer not to receive further emails from this project, reply with 'unsubscribe' and I'll remove you from the list."

  Option 2: "Reply NO to unsubscribe."

  This does not need to be formal or automated — for 8–12 emails, manual tracking of opt-outs is sufficient.

  - [ ] One opt-out line appears in every outreach email footer (even Template A and C which don't have explicit unsubscribe language in the Item 41 versions)

  **Status**: [ ] Unsubscribe language confirmed in all 3 email drafts

- [ ] **Check 1.6 — Sender email address is legitimate and deliverable**

  The from address must be a real, active email account. Spoofed or test addresses will cause bounce or spam filtering.

  Best sender options (in order of preference):
  1. Institution or organization email (highest deliverability, most credible)
  2. Custom domain email (e.g., thorn@[domain].com)
  3. Gmail or Outlook personal account (acceptable; avoid mass sending)

  SPF/DKIM check (optional but recommended if using custom domain):

  - If using Gmail/Outlook, SPF/DKIM pass is automatic — skip this check

  - [ ] From address is a real, active account (not a test or placeholder)
  - [ ] If using custom domain: SPF/DKIM verified (or confirm using Gmail/Outlook proxy)

- [ ] **Check 3.1 — Quick-reply templates ready in email drafts or GitHub templates**

  Source: `WAVE_0_CONTRIBUTOR_ONBOARDING_TEMPLATE.md` Maintainer Playbook — Response Templates

  Three response templates needed. Confirm each is saved as a draft or accessible for copy-paste:

  **Template R1: Thank you + Onboarding**
  Use when: Contributor submits a GitHub Issue (acknowledgment within 24h)

  ```
  Thank you for submitting! We have received your procedure and will complete review within 72 hours.

  If we have questions, we will ask one specific question in this issue thread.

  ```

  **Template R2: Clarification Request**
  Use when: Submission needs one specific fix before acceptance

  ```
  Thank you for submitting! We have one question before we can publish:

  [ONE SPECIFIC QUESTION — do not ask multiple questions at once]

  Once you respond, we will complete the review within 24 hours.
  ```

  **Template R3: Decline with Path Forward**
  Use when: Submission cannot be published in current form

  ```
  Thank you for submitting. We are not able to publish this procedure in its current form because:

  [SPECIFIC REASON]

  If you can [SPECIFIC FIX], we would be happy to review a revised submission.
  ```

- [ ] **Check 4.2 — Low-quality response contingency confirmed**

  Condition: More than 50% of submissions received by July 14 fail the quality gate

  Quality gate pass criteria (6 items, all must pass):
  1. Source is real, accessible, public-domain or open-licensed
  2. Procedure text matches cited source
  3. Measurements present where safety-relevant
  4. Steps are numbered and individually comprehensible
  5. License checkbox checked in submission
  6. No content requiring professional licensure

  Pre-authorized action options (no re-planning required):

  **Option 1 (preferred)**: Mark failing submissions as INTERVIEW — send a specific follow-up question (Template R2) and give the contributor 48 hours to revise. This preserves the contributor relationship.

  **Option 2 (fallback)**: If submission fails on criteria 1 or 5 (source inaccessible or license not confirmed), decline immediately with Template R3 and specific fix path. Do not attempt repair without author involvement.

  **Option 3 (if >50% of submissions are failing by July 14)**: Activate solo-content fallback. Add a note to the contributing guide clarifying the source requirement (most common failure is missing or inaccessible source). Do not increase publish volume at expense of quality gate.

  Quality gate criteria confirmed and documented in: [tracking spreadsheet link — fill in]
