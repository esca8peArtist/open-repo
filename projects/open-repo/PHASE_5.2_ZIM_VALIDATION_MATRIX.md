# Phase 5.2 ZIM Validation Matrix

## Purpose

This document answers the question: before writing any Phase 5.2 module code, are the top 5 candidates structurally compatible with the ZIM format as implemented in Phase 5.1? A compatibility failure discovered during Wave 1 implementation wastes 10–14 hours of development time. This matrix is the pre-implementation audit that prevents that failure.

The assessment criteria come directly from the Phase 5.1 constraints established in `PHASE_5.2_CAPABILITY_AUDIT.md` and the ZIM format specification from openZIM.

**Phase 5.1 ZIM constraints that all Phase 5.2 content must satisfy**:
1. Articles are static HTML only — no JavaScript, no live API calls at read time
2. All CSS must be inline in `<style>` tags; no external stylesheets
3. Images must be base64-embedded (all variant) or omitted (nopic variant)
4. Content must render correctly in Kiwix Android (WebView) and kiwix-serve (local HTTP)
5. Full-text search is Xapian-based, embedded in the ZIM — titles and body text are indexed
6. Platform is Raspberry Pi 5 aarch64 — build toolchain must produce compatible output

---

## Validation Criteria Definitions

### Criterion A: Required Metadata Fields

ZIM article metadata must include: article title, article path (URL slug), MIME type (`text/html`), and content body. Additionally, open-repo's custom metadata schema requires: `author` (source attribution), `date` (content date, not generation date), `source` (citation URL or document reference), `domain` (content taxonomy category), `version` (content version for update tracking).

Pass: All required metadata fields are present and non-null for every article generated by the module.
Fail: Any article can be generated with a null required field.

### Criterion B: Search Indexing Requirements

Xapian indexes article title and body text. For useful search results, each article must:
- Have a distinct, descriptive title (not "Article 1")
- Include all synonyms and alternate terms in either the body text or in a hidden `<meta>` tags section that the ZIM writer includes in the Xapian index
- Avoid extremely long articles (>10,000 words) that dilute search relevance

Pass: 20 randomly selected articles from a test corpus return on the first page of Kiwix search results when queried by their primary topic term.
Fail: Common queries (drug name, condition name, species name) return no results or wrong articles.

### Criterion C: Full-Text Search vs. Structured Search

ZIM supports full-text search only. There is no database query layer, no filtering by metadata field, no faceted search. A user cannot filter articles by "food type = vegetables" or "drug class = antibiotics" — they can only search text.

This is a design constraint that determines how content must be structured: category membership must be expressed in article body text, not only in metadata fields.

Pass: A user can find all articles in a category by searching a category-specific term that appears in every article in that category.
Fail: Category membership is only tracked in metadata; body text contains no category-identifying language.

### Criterion D: Multimedia Support

ZIM supports: inline HTML images (base64-encoded in articles), SVG (embedded in `<img src="data:image/svg+xml;base64,...">` or inline `<svg>` elements), HTML tables, and all standard HTML 5 presentational elements except those requiring JavaScript.

ZIM does not support: external image URLs, JavaScript-rendered charts, CSS loaded from external files, video or audio embeds.

Pass: All visual content in the module can be represented using the supported formats.
Fail: The module requires JavaScript-rendered charts, external images, or other unsupported formats to convey essential information.

### Criterion E: Version Control for Updates

When content is updated (new WHO guidelines edition, USDA canning table revision), the ZIM pipeline must regenerate articles and produce a new ZIM file. The existing Phase 5.1 pipeline handles this by running a full regeneration weekly. There is no incremental update mechanism.

Pass: All module content is regenerable from source data; no manually edited article content exists outside the source pipeline.
Fail: Any article has been hand-edited in the ZIM layer rather than in the source data, making it non-reproducible.

## Candidate 1: Medical Reference (Score 8.20) — Priority 1

### Metadata Fields Assessment: PASS

| Required field | Source | Value example | Risk |
|---|---|---|---|
| `title` | Drug INN or condition name | "Amoxicillin" / "Wound Closure" | None |
| `date` | Source document publication date | "2023-01-01" (WHO EML 23rd ed.) | Must track source version dates |
| `source` | Source document identifier | "WHO-EML-2023", "ICRC-FA-2006" | Controlled vocabulary required |
| `domain` | Fixed value | "medicine" | None |
| `version` | Incremented on source update | "1.0" | None |
| `author` | Source organization | "WHO", "ICRC", "WMS" | Multiple authors per article — use list |

**Finding**: Metadata fields are fully satisfiable. The `author` field requires list support (multiple source organizations per article). Confirm the ZIM writer accepts a list-type author field or serialize as comma-separated string.

### Search Indexing Assessment: PASS WITH REQUIREMENT

Drug monographs are highly searchable by INN name. The gap: users often know brand names, not INNs. A user searching "Tylenol" will not find the "Paracetamol" article unless aliases are indexed.

**Requirement**: Implement a `search_aliases` field in the `medical_article` schema containing brand names and common condition names. Include this field in the Xapian index by placing it in a `<meta name="keywords">` tag or in a hidden `<div>` with `display:none` CSS. Verify that Kiwix/Xapian indexes hidden text (it does — Xapian indexes all text in the HTML body regardless of CSS display property).

### Full-Text vs. Structured Search Assessment: PASS WITH DESIGN NOTE

Drug category membership (antibiotic, analgesic, antihypertensive) must appear in article body text, not only in metadata. Implementation: include a "Drug Class" field rendered as a visible line in the article header: "Class: Penicillin antibiotic". This ensures category search works without database filtering.

### Multimedia Assessment: PASS (nopic variant) / CONDITIONAL PASS (all variant)

Nopic variant: All essential clinical information is text-representable. Dosing tables, interaction tables, procedure steps — all static HTML. No clinical information is lost without images.

All variant: Injection site diagrams, wound closure technique illustrations, anatomical reference diagrams would improve usability. All of these exist as public-domain images in WHO publications and NLM MedlinePlus resources. Base64-embedding of small diagrams (~50KB each) is supported and will not cause article size issues.

**Condition**: Confirm that the largest anticipated medical article (most complex drug monograph + multiple interaction tables) stays under 200KB uncompressed. ZIM cluster compression will reduce on-disk size, but very large articles can slow rendering in Kiwix Android's WebView.

### Version Control Assessment: PASS

All medical content is sourced from versioned external documents (WHO EML editions, WMS guideline years) and internal Markdown files. The importer is fully regenerative. No manual article editing in the ZIM layer is anticipated. The `last_reviewed_date` field in the article schema tracks the most recent medical accuracy review, ensuring outdated content is flagged on regeneration.

### Overall Assessment: PASS
**Recommendation**: Proceed to Wave 1 implementation. Address the `search_aliases` requirement during stub architecture phase.

## Candidate 2: Water Systems (Score 7.90) — Priority 2

### Metadata Fields Assessment: PASS

| Required field | Source | Value example | Risk |
|---|---|---|---|
| `title` | Procedure name | "Hand Pump Construction — PVC Pitcher Pump" | None |
| `date` | Source document date | "2022-01-01" (WHO DWQG 4th ed.) | None |
| `source` | Source document identifier | "WHO-DWQG-2022", "off-grid-living/03-water.md" | None |
| `domain` | Fixed value | "water" | None |
| `version` | Incremented on update | "1.0" | None |
| `author` | Multiple sources | "WHO", "CDC", "off-grid-living" | Same list handling as Medical |

### Search Indexing Assessment: PASS

Water procedure titles are highly descriptive and directly searchable. Common user queries ("how to drill a well", "water filtration", "purify water camping") will match article content naturally. No alias system required for this domain.

**One gap**: Sizing calculations are embedded as HTML tables. A user searching "gallons per day" must find the sizing guide article. Ensure the term "gallons per day" and "liters per day" appear in the article body text (not only in table headers).

### Full-Text vs. Structured Search Assessment: PASS

Water system type (filtration, purification, well, rainwater harvesting) must appear in article body text for category search to work. Implementation: include a "System Type" field rendered in the article header.

### Multimedia Assessment: CONDITIONAL PASS

SVG schematics for pump diagrams and well cross-sections are the critical multimedia question for this module. The assessment from the Capability Audit: public-domain SVGs exist from Practica Foundation and WHO for hand pump designs.

**Validation requirement (must be confirmed before Wave 2 ZIM integration, not just Wave 1)**:
- Locate at least 3 public-domain SVG files for core water system schematics (hand pump, pitcher pump, sand filter cross-section)
- Embed one test SVG in a development ZIM article and confirm rendering in both Kiwix Android and kiwix-serve
- SVG rendering in Kiwix WebView has known edge cases: SVG elements using CSS `currentColor` or external `href` attributes may not render correctly. Test with simple inline SVG first.

If SVG rendering fails in Kiwix Android, fall back to: (1) PNG images base64-embedded in the article, or (2) ASCII art schematic with a text description. ASCII art is the reliable fallback. PNG embedded images are larger but universally supported.

### Version Control Assessment: PASS

WHO DWQG is updated on a 5-year cycle (next edition expected 2027). CDC guidance is updated irregularly. Internal Markdown sources are version-controlled in git. Full regeneration on source update is straightforward.

### Overall Assessment: CONDITIONAL PASS
**Recommendation**: Proceed to Wave 1 implementation. Validate SVG rendering in Kiwix Android before Wave 2 integration test. Have PNG fallback ready.

## Candidate 3: Seed Preservation (Score 7.80) — Priority 3

### Metadata Fields Assessment: PASS WITH NOTE

| Required field | Source | Value example | Risk |
|---|---|---|---|
| `title` | Species common name + Latin binomial | "Corn — Zea mays" | Dual-name format must be consistent |
| `date` | GRIN accession update date | "2024-06-15" | GRIN dates are per-accession; use latest accession date for species |
| `source` | GRIN accession URL or internal | "GRIN-123456", "seedwarden/seed-saving-field-manual.md" | None |
| `domain` | Fixed value | "seeds" | None |
| `version` | Incremented on update | "1.0" | None |
| `author` | USDA/internal | "USDA-GRIN", "seedwarden" | None |

**Note**: Species articles must use the format "Common Name — Genus species" in the title to ensure both search paths work (user searching "corn" and user searching "Zea mays" both return the same article).

### Search Indexing Assessment: PASS

Seed species articles are naturally searchable by common name, Latin binomial, and family name. GRIN data includes full taxonomic hierarchy — ensure family name appears in article body text so "Poaceae" returns all grass family crops.

Additional search requirement: regional synonyms. "Maize" vs. "corn", "courgette" vs. "zucchini". These must be in the search aliases or article body text.

### Full-Text vs. Structured Search Assessment: PASS WITH DESIGN NOTE

Users will want to find "all species with 5+ year viability" or "all species for Zone 4". These structured queries are not possible in Kiwix full-text search.

**Design decision**: Create category index articles. For example: "Seed Viability — Long-Term Storage (5+ years)" is an article that lists and links all species in that category. "Zone 4 — Recommended Species" is a similar index article. These index articles are pre-generated at build time from the species data. A user can then navigate to the index article and browse, rather than searching by field value.

This pattern (index articles as navigation aids) is the ZIM-idiomatic solution to the structured-search limitation. It requires 5–10 additional index articles per domain but adds zero complexity to the ZIM format itself.

### Multimedia Assessment: PASS

Seed species cards do not require diagrams to be useful. Germination calendars rendered as HTML grid tables (month across columns, species down rows) are the most useful visual element — these are pure HTML, fully compatible. Cross-links between species articles are standard ZIM internal links.

If botanical illustrations of heritage varieties are desired, apply the same Wikimedia Commons sourcing decision from Wave 3 (Botanical). Seed articles in the nopic variant are complete without images.

### Version Control Assessment: PASS

GRIN provides versioned bulk data downloads. Internal seedwarden documents are git-versioned. Full regeneration on GRIN data update is the intended workflow; no manual article editing anticipated.

### Overall Assessment: PASS
**Recommendation**: Proceed to Wave 2 implementation. Implement index articles as part of the `SeedPreservationArchiver` design.

## Candidate 4: Food Preservation (Score 7.65) — Priority 4

### Metadata Fields Assessment: PASS WITH SAFETY NOTE

| Required field | Source | Value example | Risk |
|---|---|---|---|
| `title` | Food type + preservation method | "Tomatoes — Pressure Canning" | None |
| `date` | USDA guide edition date | "2015-01-01" (2015 edition) | CRITICAL: must be exact edition year |
| `source` | USDA publication identifier | "USDA-CGOHC-2015" | Non-negotiable; cannot be blank |
| `domain` | Fixed value | "food" | None |
| `version` | Incremented on USDA update | "1.0" | Alert system needed for USDA updates |
| `author` | USDA | "USDA-NCHFP" | None |

**Safety note**: The `date` and `source` fields for food safety tables are safety-critical. If the USDA releases a new edition of the Complete Guide (the 2015 edition is the current version as of 2026), processing times may change. The ZIM must display the edition year prominently in every processing time article so users can assess currency. Implement an automated check: if the USDA NCHFP website version date differs from the bundled data date, emit a build warning.

### Search Indexing Assessment: PASS

Food preservation articles are highly searchable by food name and method. Ensure altitude adjustment terms ("high altitude canning", "pressure adjustment", "above 1000 feet") appear in relevant articles — this is a frequent search from users in mountain regions.

### Full-Text vs. Structured Search Assessment: PASS WITH INDEX ARTICLES

Same pattern as Seed Preservation: create index articles for common navigational queries:
- "Pressure Canning — All Foods" (list + links)
- "Water Bath Canning — All Foods" (list + links)
- "Fermentation — All Recipes" (list + links)

### Multimedia Assessment: PASS

Processing time tables are the core content — pure HTML tables. These are the highest-utility content in the module and require no images to be useful. Botulism risk identification boxes use the same high-contrast CSS alert pattern as the medical evacuation criteria boxes (`.danger-box` CSS class).

One multimedia consideration: jar size identification diagrams (half-pint, pint, quart, half-gallon). These are helpful but not essential — jar sizes can be described in text. Include as optional PNG in the "all" variant only.

### Version Control Assessment: PASS WITH MONITORING REQUIREMENT

The USDA Complete Guide to Home Canning has not been updated since 2015. However, a future update would invalidate the processing time tables in the ZIM — potentially causing harm if users follow outdated tables for a food type that has changed recommendations.

**Monitoring requirement**: Add a CI job that checks the USDA NCHFP "Canning" page for version date changes (compare against the stored `usda_guide_version_date` in the build configuration). If the page date changes, fail the build with a message: "USDA canning guide may have been updated — verify processing times before release." This is a one-line `curl` + `diff` check in the CI pipeline.

### Overall Assessment: PASS WITH MONITORING REQUIREMENT
**Recommendation**: Proceed to Wave 2 implementation. Implement the USDA version monitoring CI check before Wave 2 ZIM integration test.

## Candidate 5: Botanical Knowledge (Score 7.55) — Priority 5

### Metadata Fields Assessment: PASS

| Required field | Source | Value example | Risk |
|---|---|---|---|
| `title` | Common name + Latin binomial | "Wild Garlic — Allium vineale" | Consistent dual-name format required |
| `date` | USDA PLANTS last modified date | "2024-03-01" | Per-record from USDA CSV |
| `source` | USDA record ID + internal | "USDA-PLANTS-ALVI2", "seedwarden/medicinal-herbs.md" | None |
| `domain` | Fixed value | "botany" | None |
| `version` | Incremented on update | "1.0" | None |
| `author` | USDA + internal | "USDA-PLANTS", "seedwarden" | None |

### Search Indexing Assessment: PASS WITH ALIAS REQUIREMENT

Botanical search has unique requirements: users know plants by common names, which vary by region and language. "Wild garlic" in the US is not the same plant as "wild garlic" (Allium ursinum) in the UK. The USDA PLANTS database includes accepted common names by state — use the full common name list from USDA as the `search_aliases` field for each species article.

Toxicity search is the critical safety case: a user who has eaten an unknown plant and suspects poisoning must be able to search "plant identification toxic" and find the relevant article quickly. Ensure all toxic plant articles include "toxic", "poisonous", and "poisoning" in either the title or the first paragraph.

### Full-Text vs. Structured Search Assessment: PASS WITH INDEX ARTICLES

Same index article pattern as Seed and Food modules:
- "Edible Plants — Zone 4" (plants by hardiness zone)
- "Medicinal Plants — Respiratory" (plants grouped by therapeutic use)
- "Toxic Plants — High Risk Lookalikes" (the highest-priority safety index)
- "Native Plants — Midwest US" (geographic groupings)

The toxic plants index is the highest-priority index article in the botanical module. It must be findable by searching "toxic plant" and must render as a clear, scannable list with brief toxicity descriptions and links to full species articles.

### Multimedia Assessment: CONDITIONAL PASS

Botanical illustration is more important for this domain than for any other Phase 5.2 module. Plant identification without images is significantly harder. However, the ZIM format handles this well if images are sourced correctly.

**Illustration sourcing assessment**:
- Wikimedia Commons has botanical illustrations for approximately 70–80% of species in the filtered USDA PLANTS subset (food-producing and medicinal plants for US zones 3–8). Most Wikimedia Commons botanical images are CC BY or CC BY-SA licensed.
- For the nopic variant: all clinical information (edibility, toxicity, lookalike hazard) must be fully described in text. Images are not required for safety information to be communicated.
- For the all variant: include one identifying photograph (habit, flower, or leaf detail) per species where available. Base64-encode at a maximum 640px width to control article size.

**Validation requirement**: Test base64-embedded images in Kiwix Android WebView. Confirm that articles with 2–3 embedded images render in under 2 seconds on a Raspberry Pi 5 running kiwix-serve (expected: yes, given RPi 5's faster SoC vs. RPi 4).

**Lookalike hazard warning requirement**: For any species that has a toxic lookalike, the article must include a "LOOKALIKE HAZARD" alert box (using the `.danger-box` CSS class) with: (1) the lookalike species name, (2) the key distinguishing features, (3) the toxicity of the lookalike. This is a non-negotiable safety requirement. The USDA PLANTS data does not include lookalike information — this must come from seedwarden source documents or manual curation.

### Version Control Assessment: PASS

USDA PLANTS database is updated continuously; bulk CSV downloads are versioned by download date. OpenFarm archive data (CC BY 4.0) is static (no further updates since April 2025 shutdown). Internal seedwarden documents are git-versioned. Full regeneration from updated USDA CSV is the intended update workflow.

### Overall Assessment: CONDITIONAL PASS
**Recommendation**: Proceed to Wave 3 implementation. Validate base64-embedded images in Kiwix Android before the Wave 3 integration build. Confirm the Wikimedia Commons image coverage rate before committing to the "all" variant delivery in Phase 5.2.

| Water Systems | PASS | PASS | PASS | CONDITIONAL (SVG validation required) | PASS | **CONDITIONAL PASS** |
| Seed Preservation | PASS | PASS (regional synonyms) | PASS (index articles) | PASS | PASS | **PASS** |

**Both "Conditional Pass" findings are resolvable before the relevant wave begins**:
- Water Systems SVG validation: test one SVG in Kiwix Android during Wave 1 (before Wave 2 begins). PNG fallback is ready.
- Botanical image validation: test base64 image rendering during Wave 2 (before Wave 3 begins). Nopic-only release is a defined fallback.

Neither conditional finding blocks Wave 1 implementation. All five candidates are cleared for Phase 5.2 development.
